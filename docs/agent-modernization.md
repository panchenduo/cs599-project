# 智能短链 Agent 改造说明

## 选题定位

本项目围绕“企业级应用软件的 Agent 改造”展开。原系统是短链 SaaS 管理平台，已经具备完整的用户、分组、短链创建、访问统计和管理台能力。本次改造的目标是把原本依赖表单和图表的操作方式，升级为可通过自然语言驱动的智能短链运营助手。

## 改造前痛点

- 短链创建依赖表单，用户需要理解分组、有效期、描述、原始 URL 等字段。
- 查询分组、查看短链列表、创建短链和分析统计分散在不同页面。
- 统计页面展示 PV、UV、UIP 等指标，但缺少面向运营决策的自然语言解释。
- 多轮对话中“它”“这个分组”等指代依赖上下文，普通 API 调用无法天然记忆。
- 大模型调用存在失败、超时和不可用风险，需要降级和可观测设计。

## 改造后能力

- 自然语言创建短链：自动识别 URL、有效期、分组和描述。
- 分组与短链查询：通过智能助手查看当前分组列表和分组短链列表。
- 访问统计分析：复用统计服务生成 PV/UV/UIP 分析和运营建议。
- 真实 LLM Function Calling：模型根据工具 schema 自动选择工具并生成参数。
- 数据库 + Redis 记忆：长期摘要持久化，最近窗口缓存，支持多轮上下文恢复。
- 模型调度与三态熔断：候选模型失败时自动切换 fallback，保证页面可用。
- 前端可观测面板：展示工具成功率、平均耗时、意图分布、模型状态和会话记忆。

## Agent 核心流程

```text
用户自然语言输入
  -> 加载数据库摘要与 Redis 最近窗口
  -> 用户问题改写与指代消解
  -> LLM Function Calling 选择工具
  -> 调用原短链业务服务
  -> 汇总工具结果并生成回复
  -> 记录 Tool Trace 与会话记忆
  -> 返回前端可观测面板
```

## 关键工程设计

### 1. 工具边界

Agent 不直接操作数据库，而是复用原系统服务，例如 `GroupService`、`ShortLinkService`、`ShortLinkStatsService` 和 `UrlTitleService`。这样可以保留原系统的鉴权、白名单、缓存、异常处理和统计链路。

### 2. 记忆压缩

系统将会话历史拆分为长期摘要和最近窗口。数据库保存长期摘要，Redis 缓存最近窗口。最近窗口保证短期追问准确，长期摘要保留早期上下文，同时避免 Token 无限增长。

### 3. 模型降级

模型候选策略维护每个 provider 的健康状态。连续失败后进入 OPEN 状态，冷却后进入 HALF_OPEN 探测，成功则恢复 CLOSED，失败则继续熔断。safe-fallback 保证主模型不可用时仍能给出可解释响应。

### 4. 可观测性

每次工具调用都记录工具名、请求摘要、响应摘要、成功状态、耗时和错误信息。前端基于这些 trace 展示成功率、平均耗时、工具状态、意图分布和模型调度状态。

## 演示脚本

1. 打开系统管理台，进入“智能助手”页面。
2. 输入“查看当前分组列表”，展示 Agent 调用 `listGroups`。
3. 输入“查看当前分组短链列表”，展示 Agent 调用 `pageShortLink`。
4. 输入“帮我给 https://www.zhihu.com 创建一个 7 天有效的短链”，展示 Agent 自动创建短链。
5. 输入“分析默认分组最近 7 天访问情况”，展示 Agent 调用统计工具并输出运营建议。
6. 观察右侧会话记忆、模型调度状态、工具成功率和链路追踪。
